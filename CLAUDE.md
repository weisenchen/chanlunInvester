## 核心原则

### 1. 编码前思考 (Think Before Coding)

**不要假设，不要隐藏困惑，暴露权衡**

在实现缠论 v2.0 功能前：
- [ ] 明确陈述假设（如：中枢形成需要至少 3 个线段）
- [ ] 如果存在多种解释，呈现它们（如：背驰判断方法有 3 种）
- [ ] 如果有更简单的方法，说出来（如：多因子加权 vs 机器学习）
- [ ] 如果有不清楚的地方，停下来，命名困惑点，询问

**应用示例**:
```
任务：实现趋势衰减监测

思考:
1. 假设：中枢已形成，可以检测力度递减
2. 多种解释：力度可以用斜率或 MACD 面积衡量
3. 简单方法：先实现斜率判断，后续再优化
4. 困惑点：如何定义"显著"递减？需要回测确定阈值
```

---

### 2. 简单优先 (Simplicity First)

**最少代码解决问题，无投机性代码**

在缠论 v2.0 开发中：
- [ ] 不添加未要求的功能（如：先不实现市场情绪检测）
- [ ] 不为单次使用代码创建抽象（如：回测脚本先不封装）
- [ ] 不添加未请求的"灵活性"（如：权重配置先硬编码）
- [ ] 不为不可能的场景添加错误处理（如：不处理数据源永久不可用）
- [ ] 如果写了 200 行但可以用 50 行，重写它

**自问**: "资深工程师会说这过于复杂吗？" 如果是，简化。

**应用示例**:
```
❌ 过度设计:
class TrendDecayMonitor:
    def __init__(self, config_path, cache_dir, log_level):
        self.config = self._load_config(config_path)
        self.cache = CacheManager(cache_dir)
        self.logger = self._setup_logger(log_level)
        ...

✅ 简单优先:
def detect_decay(centers, segments):
    # 直接实现核心逻辑
    strengths = [calc_strength(seg) for seg in segments]
    return (strengths[-1] - strengths[-2]) / strengths[-2]
```

---

### 3. 精准修改 (Surgical Changes)

**只修改必须的，只清理自己造成的混乱**

在编辑缠论代码时：
- [ ] 不"改进"相邻代码、注释或格式
- [ ] 不重构没坏的东西
- [ ] 匹配现有风格，即使你有不同做法
- [ ] 如果发现无关的死代码，提及它，不要删除
- [ ] 只删除你的变更造成的孤儿代码（导入/变量/函数）

**测试**: 每行变更都应该直接追溯到用户需求。

**应用示例**:
```
任务：添加 buy3 检测

❌ 不相关修改:
- 修改了 buy1 检测的注释格式
- 重构了 detect_signals 函数结构
- 删除了未使用的旧函数

✅ 精准修改:
- 只添加 detect_buy3 函数
- 只在 signals 列表中追加 buy3
- 清理 detect_buy3 造成的未使用导入
```

---

### 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准，循环直到验证**

将任务转化为可验证目标：

| 任务 | 转化为目标 |
|------|-----------|
| "实现趋势衰减监测" | "编写 5 个测试用例，全部通过" |
| "修复回测脚本" | "运行回测，样本≥100 个，无错误" |
| "优化参数" | "回测胜率从 60% 提升到 65%" |

**多步骤任务计划**:
```
Phase 1 验收:
1. 运行历史数据回测 → 验证：样本≥100 个
2. 分析胜率 → 验证：≥65%
3. 分析提前天数 → 验证：≥3 天
4. 编写验收报告 → 验证：所有标准通过
```

**强成功标准**让你独立循环，**弱成功标准**（"让它工作"）需要不断澄清。

## 检查清单

### 提交前检查

- [ ] 代码是否最简单？（能更少吗？）
- [ ] 是否只修改了必要的？
- [ ] 是否有测试验证？
- [ ] 是否有文档说明？
- [ ] 是否有困惑点未解决？

### 代码审查检查

- [ ] 每行变更都追溯到需求吗？
- [ ] 有无过度抽象？
- [ ] 有无投机性代码？
- [ ] 有无未测试的功能？

## 决策记录

### 2026-04-16 - 回测方案选择

**决策**: 使用 yfinance 获取历史数据

**思考过程**:
1. 假设：yfinance 数据足够准确
2. 多种方案：yfinance vs 付费 API
3. 简单方法：先用 yfinance，后续再优化
4. 困惑点：免费数据可能有延迟，影响回测准确性

**成功标准**: 回测样本≥100 个，无数据错误

---

**这些准则生效的标志**:
- diff 中不必要的变更更少
- 因过度复杂导致的重写更少
- 澄清问题在实现之前而非错误之后
